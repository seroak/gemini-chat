## 2. 상세 분석

### 높은 사용률 Rule 분석

#### project-overview (ROI A, 효율 4.76)

- **트리거율 100%, 유용률 100%**: 모든 세션에서 프로젝트 구조 파악에 핵심 참조로 활용
- 21줄로 간결하면서도 폴더 구조, DB 스키마 등 핵심 정보를 잘 담고 있음
- **가치 평가**: alwaysApply 중 가장 높은 ROI. 현재 상태 유지가 최적

#### coding-standards (ROI B, 효율 1.96)

- **트리거율 100%, 유용률 66.7%**: 코드 작성 세션에서는 유용하나 설정/문서 작업 세션에서는 무관
- 네이밍 규칙, import 순서 등이 실질적인 코드 작성에 도움
- CSS/설정 작업(세션 1), 문서화 작업(세션 5)에서는 직접 참여도 낮음
- **가치 평가**: alwaysApply로 적절. 34줄은 합리적 크기

#### security (ROI D, 효율 0.98)

- **트리거율 100%, 유용률 16.7%**: 6개 세션 중 1개(JWT 토큰 복원 관련)에서만 유용
- alwaysApply로 매 세션 로드되지만 대부분 세션에서 보안이 직접 관련되지 않음
- 17줄로 크기는 작지만 토큰 대비 가치가 낮음
- **가치 평가**: 보안 Rule은 삭제 위험이 있으므로 적용방식 변경 검토 필요

## 4. 최적화 제안

### 즉시 실행 (ROI D 등급)

1. **security**: 적용방식을 `alwaysApply` → `globs`로 변경 검토
   - 제안 글로브: `backend/src/modules/auth/**/*.ts, backend/src/config/**/*.ts, frontend/src/services/**/*.ts`
   - 보안이 관련된 인증/설정/API 서비스 파일 작업 시에만 트리거
   - 이유: 유용률 16.7%로 대부분 세션에서 불필요한 컨텍스트 차지
   - **주의**: 보안 Rule 삭제는 위험하므로 삭제가 아닌 적용방식 변경 권장
   - **효과**: alwaysApply 총량 72줄 → 55줄 (17줄 절약, 기준 70줄 이하 달성)

2. **dev-commands**: 트리거율 0%이나 66줄로 가장 큰 Rule
   - 현재 manual로 적절하지만 참조 빈도가 0%
   - 이유: 개발 명령어/환경변수는 에디터 내에서 직접 참조하기보다 터미널에서 직접 사용
   - 제안: 내용을 핵심(pnpm 스크립트, 환경변수 목록)만 남기고 30줄 이하로 축소
   - 또는 README.md로 이관하고 Rule에서 제거
